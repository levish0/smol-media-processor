# 2026-06-22 0.2.0 Video Support + Rename

## Objective

- Add video processing to the (previously image-only) service and generalize the
  codebase. User approved: videos normalized to MP4 (H.264/AAC), FFmpeg dependency
  allowed, full generalization with clean module separation, project renamed to
  `smol-media-processor`.

## User-Approved Scope

- Output target: video → video normalization (not thumbnails).
- Output container/codec: MP4 (H.264 video, AAC audio), `+faststart`.
- FFmpeg (system binary) dependency is acceptable.
- Error handling generalized (`ImageProcessingError` → `MediaProcessingError`),
  single `/process` endpoint with auto-detection, generalized `x-media-*` headers.
- Version bumped 0.1.1 → 0.2.0; CHANGELOG updated.

## Implementation Status

- Completed: all of it (code, tests, docs, Docker, changelog, memory).
- Partial: none.
- Not started: none.

## Major Changes

- Files/modules: split `src/processor.ts` into `env.ts`, `errors.ts`, `types.ts`,
  `detect.ts`, `image.ts` (was processor.ts), `video.ts` (new), `media.ts`
  (dispatcher). Removed `processor.ts` / `processor.test.ts`.
- API / interface changes: `/process` now auto-detects image vs video via magic
  bytes and routes to `processImage` / `processVideo`. Response headers generalized
  to `x-media-*` (kind/mime-type/extension/width/height/size) plus image extras
  (`x-media-animated`, `x-media-pages`) and video extras (`x-media-duration`,
  `x-media-has-audio`). **Breaking** vs old `x-image-*`. `/health` service name is
  now `smol-media-processor`.
- Config / environment changes: new video env vars `MAX_VIDEO_INPUT_BYTES`,
  `MAX_VIDEO_OUTPUT_BYTES`, `MAX_VIDEO_DURATION_SECONDS`, `MAX_VIDEO_DIMENSION`,
  `VIDEO_TIMEOUT_SECONDS`, `VIDEO_CRF`, `VIDEO_PRESET`, `VIDEO_AUDIO_BITRATE_KBPS`.
  Image env vars unchanged. Dockerfile release stage installs `ffmpeg`.
- Tests: `image.test.ts` (migrated), `detect.test.ts` (new), `video.test.ts` (new,
  synthesizes fixtures via ffmpeg lavfi). 42 tests pass.

## Major Implementation Details

- `video.ts`: writes input to a temp file (`os.tmpdir()` + `randomUUID`), validates
  with `ffprobe` (container format whitelist + duration limit + must have a video
  stream), transcodes with `ffmpeg`, re-probes output for final dims/duration, and
  cleans up both temp files in `finally`.
- Security: ffmpeg/ffprobe invoked with `-protocol_whitelist file` (no network /
  external files); input container `format_name` is whitelisted; `-map_metadata -1`
  strips metadata; process killed on `VIDEO_TIMEOUT_SECONDS`.
- Scaling filter fits within `maxDimension`×`maxDimension` without upscaling, then
  rounds to even dims for yuv420p. Filtergraph commas are backslash-escaped (no shell).

## Remaining Work

- Known gaps: no streaming — whole file buffered in memory (same as image path).
- Risks: Elysia/Bun may impose a default request body limit below 100MB; confirm if
  large video uploads fail with a 413/parse error before raising
  `MAX_VIDEO_INPUT_BYTES` in production.
- Suggested next entry points: `src/video.ts` for codec/quality tuning;
  `src/detect.ts` to add more input container signatures.
